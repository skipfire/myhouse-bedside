ESPHome + framework esp-idf (default in recent ESPHome for ESP32)

The bedside_status component uses ESP-IDF's esp_http_client. You must expose that
built-in component to the app, or the build fails with:

  fatal error: esp_http_client.h: No such file or directory

Add under esp32 -> framework:

  advanced:
    include_builtin_idf_components:
      - esp_http_client

See ESPHome changelog 2026.2.0 and esphome/esphome#13610.

If compile fails with bedside_cjson.cpp "not found" after pulling a newer component
that moved cJSON into cjson_upstream.c, use ESPHome "Clean Build Files" once so CMake
regenerates the source list. A minimal bedside_cjson.cpp is kept in the repo to avoid
that failure on stale Ninja rules.

bedside_status: framebuffer is pushed with the same Bukys 792-wide interleaved SPI sequence as
MicroPython crowpanel579_epd.display_bitmap (not Elecrow EPD_Display). Bytes are remapped 180° vs the
Paint buffer so orientation matches the old Elecrow upload. Defaults: epd_full_refresh_on_boot true (one 0xF7 on first content frame), epd_full_refresh_every_minutes
120 (periodic full to fight thin partial ink), epd_partial_passes 2 between full refreshes. Set every_minutes
to 0 to drop scheduled full flashes (thinner text). Removed epd_full_update (full on every draw).
If YAML says that option is invalid, the add-on is using an old copy of the external
component — set external_components refresh (e.g. 1h) or reinstall/update the git ref.
